# 07 - 开发路线图：从当前固件到最终产品形态

> 本文与 [00-愿景](00-vision.md)、[01-功能性需求](01-requirements-functional.md)、[02-非功能需求](02-requirements-nonfunctional.md)、[03-硬件设计](03-hardware-design.md)、[04-固件架构](04-firmware-architecture.md)、[05-云与 App](05-cloud-and-app.md)、[06-项目执行清单](06-project-checklist.md) 对齐。  
> **执行方式**：按阶段顺序推进；每阶段末尾有可验收条目。硬件未到货处可用 **mock 接口** 占位，避免阻塞 UI 与云端。

---

## 最终产品形态（一句话）

可日常佩戴的儿童/家庭手表：**稳定圆屏与触控、可信时间、合理续航、传感器与业务数据闭环、与云端及手机 App 协同、可 OTA 升级**，并满足文档中的安全与隐私边界。

---

## 阶段 0：基线与工程纪律

**目标**：任何人可复现构建烧录；引脚与板级差异可配置；串口日志可诊断。

| 步骤 | 内容 | 验收 |
|------|------|------|
| 0.1 | 固定 ESP-IDF 版本、目标 `esp32s3`、Flash 容量与 `sdkconfig` / `sdkconfig.defaults` 一致 | 无「Detected size larger than header」类误导性告警或已逐项解释并记录 |
| 0.2 | `firmware/main` 拆目录：`board/`（BSP+引脚）、`ui/`、`services/`、`app/` 启动编排 | `app_main` 仅负责启动与依赖注入，业务不在单文件堆叠 |
| 0.3 | 分区表与后续 OTA 策略预留（可先占位） | 文档或注释中写明当前分区用途 |
| 0.4 | 构建与烧录说明与仓库一致（命令行为主） | `idf.py build` / `flash` / `monitor` 路径在 README 或 06 中可找到 |

**阶段 0 执行记录**

- **0.1（已完成，2026-05-12）**：`CONFIG_ESPTOOLPY_FLASHSIZE_16MB` 已写入 `firmware/sdkconfig.defaults` 与仓库内 `firmware/sdkconfig`，与 N16 板载 16MB Flash 一致，消除「Detected size larger than binary header」类告警。若你本地只改了 defaults，可执行 `idf.py reconfigure` 让 Kconfig 合并。
- **0.2（已完成，2026-05-12）**：`firmware/main/` 已拆为 `board/`（`bsp_board.h`、`bsp_lcd.*`）、`ui/`（`ui_demo.*`）、`services/`（`svc_system.*`，含 NVS 早期初始化）、`app/`（`watch_app.*`）；`app_main.c` 仅调用 `watch_app_start()`。
- **0.3（已完成，2026-05-12）**：新增 **`firmware/partitions_watch.csv`**（`nvs`、`phy_init`、`factory` + **`otadata` / `ota_0` / `ota_1`** 占位，各 app 槽 **2 MiB**）；`sdkconfig` 与 **`sdkconfig.defaults`** 启用 **`CONFIG_PARTITION_TABLE_CUSTOM`** 并指向该文件；用途与 OTA 预留策略见 **CSV 顶部注释** 与 **`firmware/README.md`** 内「分区表（阶段 0.3）」一节。
- **0.4（已完成，2026-05-12）**：**`idf.py set-target` / `build` / `flash` / `monitor`** 的路径与命令表写入 **`firmware/README.md`**（「命令速查」「烧录与监视」），并在 **[06-project-checklist.md](06-project-checklist.md) 的 P1.0** 增加同口径检索段落。

**与 06 的关系**：对应 P0～P1 中的工具链与「固件能跑」部分；详细打勾仍以 [06](06-project-checklist.md) 为准。

## 阶段 1：能当表用（时间 + 表盘壳 + 省电雏形）

**目标**：用户抬腕或操作后，**主表盘显示可信时间**；可进入设置再返回；具备基本亮屏/灭屏策略。

| 步骤 | 内容 | 验收 |
|------|------|------|
| 1.1 | LVGL 多 `screen`：表盘、设置列表、关于/版本 | 三屏可切换，无内存泄漏（粗测：反复切换 30 分钟） |
| 1.2 | **RTC（PCF85063）** 读时间；时区策略先简后全（可先固定东八区） | 断电后有电池时时间保持；误差在规格内见 03 |
| 1.3 | （可选增强）WiFi SNTP 校时，与 RTC 写回策略 | 校时成功后表盘时间跳变可接受且有日志 |
| 1.4 | 电量显示占位 → 接 **BAT_ADC** 与电量模型 | 表盘显示 0–100% 或档位，低电量有提示 |
| 1.5 | 背光/亮屏时长、空闲灭屏；与 LVGL 任务协调 | 静置达到阈值灭屏；触摸或按键唤醒 |

**依赖**：03 中 GPIO map；04 中任务与电源状态机。

**阶段 1 执行记录**

- **1.1（已完成，2026-05-12）**：`ui_demo.c` 使用 **`lv_obj_create(NULL)`** 建立 **表盘、设置（`lv_list`）、关于** 三屏；**`lv_scr_load`** 切换并 **`lv_obj_del`** 默认首屏；`esp_app_get_description()` 显示固件版本与 IDF 串；表盘时间为占位（**1.2 RTC**）。Kconfig 启用 **SimSun 16 CJK** 为默认字体、**Montserrat 48** 仅用于大号时间。粗测防泄漏：切换路径不 `new` 新 screen，仅重复 `load`。
- **1.2（已完成，2026-05-14）**：**`svc_rtc.*`** 经共享 **I2C0** 读 **PCF85063A（7-bit `0x51`）**；上电清除 **CTRL1.STOP**；**`lv_timer`** 每秒刷新表盘 **HH:MM** 与 **YYYY-MM-DD**。**时区**：当前将芯片寄存器视为 **东八区墙钟**（无夏令时；与 1.3 SNTP 写入策略衔接时再细化 UTC/本地）。无应答或 **OS** 位时表盘降级为 **`--:--`** 与状态文案。
- **1.3（已完成，2026-05-14）**：**`svc_timesync_wifi.*`**：`esp_netif` + **STA**；**`pool.ntp.org`** SNTP；**`TZ=CST-8`** 后 **`localtime_r`** 得到墙钟并 **`svc_rtc_write_local`** 写回 PCF85063；写 RTC 前 **`lvgl_port_lock`** 避免与触摸共用 I2C 交错。凭据在 **menuconfig → Watch WiFi / SNTP**；**SSID 为空则跳过**联网。成功/失败均有 **`svc_ts` / `svc_rtc` 日志**。

## 阶段 2：传感器与本地数据

**目标**：腕上数据本地可用，为运动与健康、云端上报打底。

| 步骤 | 内容 | 验收 |
|------|------|------|
| 2.1 | **QMI8658** 驱动与数据通路（I2C、中断脚） | 能读原始数据或简化步数；不阻塞 UI 线程 |
| 2.2 | 表盘或子页展示步数/活动状态（可先简化算法） | 与参考实现或人工计数对比数量级合理 |
| 2.3 | **NVS** 存用户偏好（表盘、音量、开关等） | 重启后配置保持 |
| 2.4 | （按需）LittleFS/SPIFFS 存表盘资源或缓存 | 挂载失败有降级 UI |

## 阶段 3：连接与协议闭环（蜂窝 + 后端）

**目标**：设备具备**稳定上行/下行**一条路径（先心跳与配置，再业务）。

| 步骤 | 内容 | 验收 |
|------|------|------|
| 3.1 | **EC800** UART AT 封装：注册、PDP、TLS/MQTT 最小集 | 与 06 中 P0.4～P0.5 清单一致或可引用其结果 |
| 3.2 | 固件侧 MQTT/HTTPS 与 [05](05-cloud-and-app.md) topic/API 对齐 | 心跳上报；云端可查设备在线 |
| 3.3 | 下行修改一项简单参数（如上报间隔） | 设备端持久化并生效 |
| 3.4 | 位置/GNSS 或 LBS 回退策略（与 05 数据模型一致） | 端到端一条轨迹或点位可在后端/App 看到 |

**与 06 的关系**：对应 P1「固件原型」与 P3「端到端联调」主体。

## 阶段 4：手机 App 与账号

**目标**：用户可绑定设备、查看核心数据、接收关键通知（占位到真推送）。

| 步骤 | 内容 | 验收 |
|------|------|------|
| 4.1 | 账号体系、设备绑定（码/二维码） | 新用户完成绑定后设备列表可见 |
| 4.2 | 地图/轨迹/微聊等按 01 优先级分期 | 与 P3 里程碑对齐 |

## 阶段 5：完整体验与表业特性

**目标**：接近商品化交互与可靠性。

| 步骤 | 内容 | 验收 |
|------|------|------|
| 5.1 | 表盘主题、通知、闹钟、勿扰、运动记录等按 01 分期 | 每功能有验收用例（可链到 01 FR 编号） |
| 5.2 | **OTA**：签名、分区、失败回滚 | 失败可恢复；版本号可追溯 |
| 5.3 | 安全：TLS、凭据存储、日志脱敏（对齐 02） | 自检清单或第三方粗测通过 |
| 5.4 | 压力与长稳：典型场景 + coredump/日志策略（见 04） | 记录一次 24h～72h 结果入 06 或单独测试记录 |

**目标**：从开发板形态迁移到自研硬件与表壳，工厂可生产。

| 步骤 | 内容 | 验收 |
|------|------|------|

---

## 推荐默认优先级（若资源有限）

1. **阶段 1**（表 + 时间 + 省电）— 用户可感知价值最大。  
2. **阶段 3** 中与运营商无关部分：**mock 网络** 驱动 UI 与云端接口。  
3. **阶段 2** 与 **阶段 3** 在人力允许时并行（传感器不依赖公网）。  
4. **阶段 4** 与固件阶段 3 后半并行。
