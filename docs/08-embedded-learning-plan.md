# 08 - 嵌入式概念学习计划（面向 watch-diy 项目）

> 适用对象：从零开始学习嵌入式，但目标不是独立手写完整驱动，而是能深入理解项目里的硬件、接口、日志、调试现象和技术决策。  
> 对齐文档：[03-硬件设计](03-hardware-design.md)、[04-固件架构](04-firmware-architecture.md)、[06-项目执行清单](06-project-checklist.md)、[07-开发路线图](07-development-roadmap.md)。  
> 项目硬件主线：ESP32-S3 圆屏开发板 + EC800 系列 4G/GNSS 板，后续自研 PCB 使用 AMOLED、触摸、IMU、RTC、电量计、I2S 音频、摄像头、电源管理等外设。

---

## 1. 学习目标

完成本计划后，应能做到：

- 看懂本项目系统框图、电源树、GPIO 分配表，以及「哪个外设通过什么总线连接到 ESP32-S3」。
- 理解常见嵌入式概念：DIO/GPIO、ADC、PWM、UART、I2C、SPI/QSPI、I2S、DMA、中断、上拉/下拉、电平、共地、低功耗、AT 指令、GNSS、蜂窝 PDP/MQTT/TLS。
- 能阅读芯片数据手册里的关键表格：供电电压、接口电平、引脚方向、时序、I2C 地址、UART 波特率、电流峰值。
- 能用串口工具、万用表、USB 电流表和项目日志解释常见问题：无响应、乱码、I2C 设备不应答、ADC 数值跳、4G 掉线、GNSS 无 fix、屏幕不亮、休眠不生效。
- 能参与 P0～P1 阶段的硬件验证和问题定位，即使不亲自编写底层驱动，也知道软件为什么这样设计。

不以这些为目标：

- 不要求从零写 ESP-IDF 驱动。
- 不要求手写 LVGL、MQTT、TLS、Opus、摄像头完整实现。
- 不要求成为 PCB 硬件工程师，但要能理解原理图和关键风险。

---

## 2. 学习方式

建议周期：**8 周**。每周 4～6 小时即可，重点是“概念 + 观察 + 记录”。

这份计划不要求你一开始就能看懂完整原理图。每个主题都先给一个**已经简化过的项目版本**，你先照着抄、照着标注，再逐步加细节。

每个主题都按四步走：

1. **先看项目对应点**：从本项目某个模块出发。
2. **再学通用概念**：理解接口和电气规律。
3. **做轻量实验**：以观察、测量、读日志为主，不强求写代码。
4. **写一页记录**：用自己的话解释现象，沉淀到项目文档。

推荐记录位置：

- P0 硬件验证：`hardware/docs/p0-log.md`
- 功耗模型：`hardware/docs/power-model.md`
- 学习笔记：可新建 `docs/embedded-notes/`

### 2.1 推荐准备的小工具

有实物硬件后，建议按优先级准备这些工具。不需要一次买齐，先用已有设备做低门槛 demo。

| 工具 | 优先级 | 用来观察什么 | 备注 |
|------|--------|--------------|------|
| 串口调试软件 | 必备 | AT 文本、ESP32 日志、URC | PuTTY、MobaXterm、友善串口助手等 |
| 万用表 | 必备 | GND 是否共地、电压高低、电池电压 | 不适合观察高速通信波形 |
| USB 电流表 | 推荐 | 亮屏、4G、GNSS、传数据时的电流变化 | 对理解续航很有帮助 |
| 8 通道 USB 逻辑分析仪 | 推荐 | UART、I2C、SPI/I2S 的数字波形和协议解码 | 常见 24MHz 款即可入门 |
| 示波器 | 可选 | 电平质量、边沿、纹波、4G 供电跌落 | 没有也不影响本计划大部分学习 |

安全边界：

- 只观测 **GND、3.3V 逻辑信号、USB 5V、电池电压** 这些明确位置。
- 不要把 5V 接到 ESP32-S3 的 GPIO。
- 不要用万用表电流档直接跨接电池正负极。
- 不要随意碰 4G/GNSS 天线馈线、RF 匹配网络和 EC800 的 VBAT_RF 大电流路径。
- 逻辑分析仪通常只需要接 **GND + 信号线**，不要把分析仪的 VCC 当供电线乱接。

### 3.2 第二眼再看“它们用什么线说话”

```text
                             │

│                    ESP32-S3                         │
│ 主控 MCU：运行 ESP-IDF / FreeRTOS / LVGL / 业务逻辑 │
└──────┬────────┬────────┬────────┬────────┬────────┘
       │        │        │        │        │
       │QSPI    │I2C     │UART    │I2S     │GPIO/ADC/PWM
       ▼        ▼        ▼        ▼        ▼
┌─────────┐ ┌─────────┐ ┌────────────┐ ┌─────────┐ ┌────────────┐
│ AMOLED  │ │ 触摸/IMU │ │ EC800 4G   │ │ 麦克风/  │ │ 按键/电池/ │
│ 屏幕     │ │ RTC/电量 │ │ GNSS 模组  │ │ 扬声器   │ │ 振动马达   │
└─────────┘ └─────────┘ └────────────┘ └─────────┘ └────────────┘
```

这一层开始记接口名：

- **QSPI**：给屏幕快速传大量像素数据。
- **I2C**：连接触摸、IMU、RTC、电量计这类低速外设。
- **UART**：连接 EC800，用 AT 指令控制 4G/GNSS。
- **I2S**：传音频数据，连接麦克风和功放。
- **GPIO**：按键、中断、复位、开关控制。
- **ADC**：测电池电压这类模拟量。
- **PWM**：控制振动马达这类“强弱/节奏”。

## 6. S0：嵌入式系统全景

### 目标

建立一张“手表里面谁和谁说话”的地图。

### 项目切入点

阅读 [03-硬件设计 §1](03-hardware-design.md) 的系统框图和 [04-固件架构 §1](04-firmware-architecture.md) 的软件分层。

### 需要理解的概念

- MCU：ESP32-S3 是主控，负责 UI、外设、业务逻辑。
- 模组：EC800 是蜂窝/GNSS 模组，内部已经实现大量通信协议，ESP32 主要通过 UART AT 指令控制它。
- 外设：屏幕、触摸、IMU、RTC、电量计、麦克风、功放、摄像头等都不是“软件功能”，而是通过不同总线挂在主控上。
- 驱动：把“总线读写”包装成“读时间、读电量、画屏幕、采音频”的软件层。
- 服务：在驱动之上做定位、通信、音频、电源管理等业务。

### 轻量实验

不要直接从完整硬件图开始。按下面 4 步做：

1. **抄图**：把 “3.1 第一眼只看 5 个框” 抄到纸上。
2. **补接口**：把 “3.2 第二眼再看接口” 抄一遍，重点写出 QSPI、I2C、UART、I2S、GPIO、ADC、PWM。
3. **填模板**：完成 “3.4 你要画的第一张图长这样” 里的填空。
4. **对照完整图**：打开 [03-硬件设计 §1](03-hardware-design.md)，只找你已经认识的词：ESP32-S3、EC800、AMOLED、I2C、UART、I2S、GPIO。先不要管看不懂的芯片型号。

完成后，你的输出不需要漂亮，只要像这样即可：

```text
ESP32-S3 是主控。
屏幕走 QSPI，因为像素数据多。
触摸、IMU、RTC、电量计走 I2C，因为它们数据少，可以共享两根线。
EC800 走 UART，因为 ESP32 用 AT 指令控制它。
麦克风和扬声器走 I2S，因为音频是连续数据流。
按键、复位、中断走 GPIO，因为只需要高低电平。
电池电压走 ADC，因为它是模拟电压。
振动马达走 PWM，因为要控制振动节奏。
```

### 验收问题

- 为什么 EC800 不直接叫“传感器”，而叫“模组”？
- 为什么触摸、IMU、RTC、电量计可以共享一组 I2C？
- 为什么屏幕和音频需要更高吞吐的数据通路？

## 7. S1：电气基础、GPIO 与 DIO

### 目标

理解数字输入输出是嵌入式最基础的“开关语言”。

### 项目切入点

阅读 [03-硬件设计 §4](03-hardware-design.md) GPIO 资源分配，重点看：

- `SOS_KEY`、`POWER_KEY`
- `LTE_PWRKEY`、`LTE_RST`
- `TP_INT`、`IMU_INT`
- `LTE_STATUS`、`LTE_RI`

### 需要理解的概念

- **DIO/GPIO**：数字输入输出，只有高/低两个逻辑状态。
- **输入**：按键、中断脚、状态脚。MCU 读取外部电平。
- **输出**：复位、开关、片选、PWRKEY。MCU 主动拉高或拉低。
- **高电平/低电平**：本项目多数逻辑电平是 3.3V，不要把 5V 直接接到 ESP32 GPIO。
- **共地 GND**：两个板子之间要通信，必须共享参考地。
- **上拉/下拉**：让输入脚在没人按、没人驱动时也有确定状态。
- **中断 INT**：外设主动提醒 MCU“我有事”，避免 MCU 一直轮询。
- **开漏/推挽**：理解即可，重点知道 I2C 常用开漏 + 上拉。

### 轻量实验

- 用万用表测开发板 GND 与蜂窝板 GND 是否连通。
- 按下/松开按键时，观察日志或万用表电平变化。
- 查 EC800 硬件手册中 PWRKEY 的有效电平和保持时间，对照 [04-固件架构 §4](04-firmware-architecture.md) 启动流程里的“PWRKEY 拉低 800ms”。

### 上手 demo

**Demo S1-1：确认两块板“共地”**

目的：理解为什么 UART/I2C/SPI 这类信号都需要共同参考点。

准备：万用表、ESP32-S3 开发板、EC800 开发板。

步骤：

1. 两块板都断电。
2. 万用表拨到蜂鸣档或电阻档。
3. 一支表笔碰 ESP32-S3 板的 GND，另一支碰 EC800 板的 GND。
4. 如果已经用杜邦线把两块板 GND 连在一起，应听到蜂鸣或看到接近 0Ω。
5. 记录结果：`ESP32 GND ↔ EC800 GND：导通 / 不导通`。

你应该理解到：

- “高电平 3.3V”不是绝对概念，是相对 GND 的电压。
- 没有共地时，ESP32 发出的 UART 高低电平，EC800 可能无法正确判断。

**Demo S1-2：观察一个按键输入**

目的：理解 GPIO 输入、上拉/下拉、按下/松开的状态变化。

步骤：

1. 找到圆屏板上的 BOOT、Home 或可用按键。
2. 用 `idf.py monitor` 或当前固件日志观察按键事件；如果固件暂时没有按键日志，就先只看硬件文档中的 GPIO 分配。
3. 用万用表电压档测按键 GPIO 对 GND 的电压，分别记录按下和松开时的电压。

记录模板：

```text
按键名称：
```

## 8. S2：UART、串口日志与 AT 指令

### 目标

能看懂 ESP32 与 EC800 的基本通信方式，理解 P0 验证里的串口调试。

### 项目切入点

阅读 [06-项目执行清单 §P0.2～P0.5](06-project-checklist.md)，重点看：

- 串口工具、COM 口、115200 波特率
- `AT` / `OK`
- `+` 开头的 URC
- PDP、MQTT/TLS、GNSS 验证

### 需要理解的概念

- **UART**：异步串行通信，常见线包括 TX、RX、GND，可选 RTS/CTS 流控。
- **波特率**：双方说话速度必须一致，常见 115200。
- **TX/RX 交叉**：发送接对方接收。
- **乱码**：常见原因是波特率错、电平不匹配、GND 没共地、线不稳。
- **AT 指令**：主控给蜂窝模组发送文本命令，模组返回文本响应。
- **URC**：Unsolicited Result Code，模组主动上报的异步消息，例如网络状态变化、来电、定位结果。
- **流控 RTS/CTS**：数据量较大或模组忙时，用硬件信号告诉对方暂停/继续发送。

### 轻量实验

- 使用串口工具连接 EC800 板，手动发送 `AT`，观察 `OK`。
- 发送 SIM、信号、注册相关查询命令，以实际模组 AT 手册为准。
- 连续打开串口 5 分钟，记录是否乱码、断线、出现 URC。
- 把一次完整 AT 交互贴到 `hardware/docs/p0-log.md`，标注“命令、响应、主动上报”。

### 上手 demo

#### Demo S2-1：先不接 ESP32，直接和 EC800 说话

目的：确认 EC800 模组本身、SIM、串口工具、波特率是可用的。

1. EC800 开发板插电脑。
2. 在系统设备管理器或串口工具里找到新增 COM 口。
3. 串口参数先用：`115200, 8N1, no flow control`。
4. 发送 `AT`，期望看到 `OK`。
5. 发送 `ATI` 或模组手册推荐的版本查询命令，记录返回内容。
6. 保持串口打开 5 分钟，观察是否出现 `+` 开头的主动上报。

#### Demo S2-2：理解 ESP32-S3 和 EC800 的 UART 接线

目的：把“TX 接 RX”从文字变成你能在板子上指出来的线。

本项目设计关系：

```text
ESP32-S3 GPIO21  LTE_UART_TX  ───→  EC800 RX
ESP32-S3 GPIO47  LTE_UART_RX  ←───  EC800 TX
ESP32-S3 GPIO48  LTE_UART_RTS ───→  EC800 CTS   （可选硬件流控）
ESP32-S3 GPIO45  LTE_UART_CTS ←───  EC800 RTS   （可选硬件流控）
ESP32-S3 GND                  ─────  EC800 GND
```

如果你用的是开发板/评估板，实际排针名字可能不是 `GPIO21/GPIO47`，而是 `TXD/RXD`、`UART_TX/UART_RX`、`MCU_TX/MCU_RX`。以你手头板子的丝印和手册为准。

关键判断：

- ESP32 的 **TX** 是“ESP32 发出去”，所以要接 EC800 的 **RX**。
- ESP32 的 **RX** 是“ESP32 收进来”，所以要接 EC800 的 **TX**。
- GND 必须连。
- 先不要接 5V 到任何 UART 信号线上。

#### Demo S2-3：用逻辑分析仪观察 UART 波形

目的：真正看到 UART 不是“神秘文本”，而是一串高低电平脉冲。

准备：

- USB 逻辑分析仪。
- 逻辑分析软件，如 PulseView / Saleae Logic。
- ESP32-S3 与 EC800 已按 UART 方式连接，或 EC800 开发板已有可观测的 TX/RX 排针。

推荐夹线：

```text
逻辑分析仪 GND  → 板子 GND
逻辑分析仪 CH0  → ESP32 TX / EC800 RX 这根线
逻辑分析仪 CH1  → ESP32 RX / EC800 TX 这根线
```

不要夹：

- 不要夹 EC800 天线座附近的 RF 线。
- 不要夹 VBAT_RF 大电流供电脚。
- 不要把逻辑分析仪的 VCC 接到不确定的板上电源。

软件设置：

| 项目 | 建议值 |
|------|--------|
| 采样率 | 1MHz 以上，推荐 4MHz 或 8MHz |
| 协议解码 | Async Serial / UART |
| 波特率 | 115200 |
| 数据位 | 8 |
| 校验 | None |
| 停止位 | 1 |
| 电平 | UART 空闲通常为高电平 |

操作步骤：

1. 先打开逻辑分析软件，开始采样。
2. 在串口工具或 ESP32 固件里触发一次 `AT` 发送。
3. 停止采样。
4. 给 CH0、CH1 分别加 UART 解码器。
5. 找到 ASCII 内容：一边应该能看到 `AT`，另一边应该能看到 `OK` 或模组返回。

你应该观察到：

- UART 空闲时通常保持高电平。
- 每个字节开始前有一个低电平 start bit。
- 115200 波特率下，一个 bit 约 `1 / 115200 = 8.68us`。
- 如果解码波特率设置错，软件会显示乱码。
- 如果只看到一边有波形，另一边没有，可能是对方没回、线接错、没共地、模组没开机或波特率不对。

记录模板：

```text

CH1 接在：
采样率：

CH0 解码内容：
CH1 解码内容：
是否看到 AT：
是否看到 OK：
遇到的问题：
```

#### Demo S2-4：故意把“解码波特率”调错

目的：理解乱码不一定是线坏，也可能是双方速度理解不一致。

## 9. S3：I2C 总线

### 目标

理解低速外设为什么常用 I2C，以及共享总线的好处和风险。

### 项目切入点

阅读 [03-硬件设计 §1](03-hardware-design.md) 中 I2C 外设：

- FT3168 触摸
- QMI8658 IMU
- PCF85063 RTC
- BQ27220 电量计
- OV2640 的 SCCB，可近似按 I2C 思路理解

### 需要理解的概念

- **SCL/SDA**：I2C 只有时钟线和数据线，两根线可挂多个设备。
- **主从结构**：ESP32 通常是 master，外设是 slave。
- **7-bit 地址**：同一总线上的设备靠地址区分。
- **ACK/NACK**：设备是否确认收到。
- **上拉电阻**：I2C 依赖上拉形成高电平。
- **总线占用**：多个任务访问同一 I2C 时要加锁，否则读写可能交错。
- **速率**：100kHz、400kHz 常见；线长、电容、上拉都会影响稳定性。

### 轻量实验

- 查每个 I2C 外设的数据手册，记录默认地址。
- 对照固件或日志，观察 RTC 读时间、触摸中断、IMU 读数是否都经过同一组 `GPIO15/GPIO16`。
- 如果项目已有 I2C 扫描日志，学习如何从扫描结果判断设备是否在线。

操作步骤：

## 10. S4：SPI/QSPI、显示与大数据搬运

### 目标

理解屏幕为什么不像按键那样简单，以及帧缓冲、DMA、PSRAM 为什么重要。

### 项目切入点

阅读：

- [03-硬件设计 §1](03-hardware-design.md) AMOLED `SH8601 / QSPI`
- [03-硬件设计 §4](03-hardware-design.md) `LCD_CS/SCK/D0-D3/RST/TE`
- [04-固件架构 §3](04-firmware-architecture.md) LVGL 帧缓冲、PSRAM、DMA

### 需要理解的概念

- **SPI**：同步串行总线，常见有 SCK、MOSI、MISO、CS。
- **QSPI**：用 4 条数据线并行传输，比普通 SPI 更适合屏幕或 Flash。
- **CS 片选**：告诉哪个设备正在通信。
- **RST**：硬件复位屏幕。
- **TE**：Tearing Effect，同步刷新，减少撕裂。
- **帧缓冲**：屏幕图像在内存里的像素数据。
- **RGB565**：一种 16-bit 像素格式；466×466×2 字节约 424KB，双缓冲约 848KB。
- **DMA**：外设直接搬运内存数据，减少 CPU 逐字节搬运压力。
- **PSRAM**：外部大内存，适合放大缓冲，但 DMA 能力和速度要注意。

### 轻量实验

- 计算一次全屏 RGB565 刷新的字节数：`466 × 466 × 2`。
- 观察 LVGL 表盘切换、动画、亮灭屏时的流畅度和功耗变化。
- 对照日志或配置，理解为什么项目关心 `min free PSRAM`。

### 上手 demo

**Demo S4-1：不用探针，先用计算理解屏幕数据量**

目的：知道为什么屏幕需要 QSPI、帧缓冲和 DMA。

步骤：

1. 计算全屏像素数：`466 × 466 = 217156`。
2. RGB565 每个像素 2 字节，所以一帧约 `434312` 字节，约 `424KB`。
3. 如果 30 FPS 全屏刷新，理论数据量约 `424KB × 30 = 12.4MB/s`。
4. 对比 UART 115200：每秒大约只有 11KB 量级有效数据。

你应该理解到：

- 屏幕数据量远大于 AT 指令文本。
- QSPI、DMA、PSRAM 是为“大量连续数据”服务的。

## 11. S5：ADC、PWM、电池与功耗

### 目标

理解模拟量、占空比、电池电压、电流和续航之间的关系。

### 项目切入点

阅读：

- [07-开发路线图 §阶段 1.4](07-development-roadmap.md) `BAT_ADC`
- [06-项目执行清单 §P0.7](06-project-checklist.md) 功耗与热
- [03-硬件设计 §3](03-hardware-design.md) 电源树
- [03-硬件设计 §4](03-hardware-design.md) `VIB_PWM`

### 需要理解的概念

- **ADC**：把模拟电压转换成数字值。
- **分压**：电池电压可能超过 ADC 安全范围，需要电阻分压后测量。
- **校准与误差**：ADC 不是理想电压表，会受参考电压、噪声、分压电阻误差影响。
- **PWM**：用快速开关的占空比控制平均能量，常用于振动马达、背光、蜂鸣器等。
- **mA / mAh**：电流和容量不是一回事；500mAh 电池在 50mA 平均电流下理论约 10 小时。
- **峰值电流**：4G 发射瞬间可能需要很高电流，平均功耗低不代表电源设计安全。
- **LDO 与直供**：EC800 直接吃电池 VSYS，避免 LDO 压降和发热。

### 轻量实验

- 用 USB 电流表观察亮屏、息屏、4G 注册、传数据、GNSS 搜星时电流变化。
- 用万用表测电池电压，对比 UI 中电量百分比。
- 在 `hardware/docs/power-model.md` 建一个简表：状态、估算电流、每天持续时间、mAh 消耗。

## 12. S6：I2S、音频与实时流

### 目标

理解麦克风、功放、PCM/Opus 语音链路为什么比普通传感器更强调实时性和缓冲。

### 项目切入点

阅读：

- [03-硬件设计 §1](03-hardware-design.md) I2S 麦克风、I2S 功放、EC800 PCM
- [04-固件架构 §2](04-firmware-architecture.md) `audio_task`
- [04-固件架构 §3.2](04-firmware-architecture.md) 音频/Opus 内存预算
- [06-项目执行清单 §P1.4](06-project-checklist.md) 音频路径

### 需要理解的概念

- **I2S**：音频专用串行总线，典型线包括 BCLK、WS/LRCLK、DATA。
- **采样率**：每秒采多少个声音样本，例如 16kHz。
- **位宽**：每个样本多少 bit，例如 16-bit、24-bit。
- **PCM**：未压缩音频数据。
- **Opus**：语音压缩编码，减少网络上传数据量。
- **环形缓冲**：采集和编码速度不完全一致时，用缓冲吸收抖动。
- **延迟**：缓冲越大越稳，但实时通话/微聊延迟越高。

### 轻量实验

- 计算 16kHz、16-bit、单声道 PCM 每秒数据量：`16000 × 2 = 32KB/s`。
- 对比一段 PCM 原始数据和 Opus 压缩后大小，理解为什么网络语音要编码。
- 观察录音上传时是否影响定位、MQTT、UI 帧率，对照 [04-固件架构 §3.5](04-firmware-architecture.md) 资源优先级。

### 上手 demo

**Demo S6-1：用逻辑分析仪看 I2S 三根线**

目的：区分 I2S 和 I2C，看到音频是连续数据流。

推荐夹线：

```text
逻辑分析仪 GND  → 板子 GND
逻辑分析仪 CH0  → BCLK
逻辑分析仪 CH1  → WS / LRCLK
逻辑分析仪 CH2  → DATA
```

操作步骤：

1. 触发录音或播放功能。
2. 开始采样。
3. 观察 BCLK 是否连续跳变，WS 是否按左右声道/帧节奏变化，DATA 是否持续有数据。

你应该理解到：

- I2S 不是偶尔读一个寄存器，而是连续搬运音频样本。
- 没有录音/播放时，I2S 线可能停止或保持空闲。
- 音频越实时，越依赖 DMA、缓冲和任务调度。

如果你没有麦克风/功放小板或当前固件还没接音频，这个 demo 延后到 P1.4。

### 验收问题

- I2S 和 I2C 名字像，但解决的问题有什么不同？
- 为什么音频需要 DMA 和环形缓冲？
- 为什么录音上传时 GNSS 定时采集可能要让路？
- PCM 与 Opus 的关系是什么？

## 13. S7：蜂窝网络、PDP、MQTT/TLS 与 GNSS

### 目标

理解设备如何通过 4G 到云端，以及定位数据从哪里来。

### 项目切入点

阅读：

- [06-项目执行清单 §P0.4～P0.6](06-project-checklist.md)
- [06-项目执行清单 §P1.2～P1.3](06-project-checklist.md)
- [04-固件架构 §5.3](04-firmware-architecture.md) MQTT 连接状态机附近内容

### 需要理解的概念

- **SIM / ICCID / IMEI**：卡身份与设备身份。
- **注册网络**：模组先接入运营商网络，才可能上网。
- **信号强度**：影响连接稳定性、功耗、定位和传输时延。
- **PDP Context**：蜂窝数据承载，类似“拿到可用的数据通道”。
- **TCP/TLS**：TCP 提供连接，TLS 提供加密和身份校验。
- **MQTT**：轻量消息协议，适合设备心跳、上报、下行命令。
- **HTTPS**：适合配置、文件上传、语音片段上传等。
- **GNSS**：卫星定位，冷启动、热启动、开阔天空、室内无 fix 都会影响结果。
- **NMEA**：常见 GNSS 文本数据格式。
- **LBS 回退**：无 GNSS 时可用基站/网络定位做低精度兜底。

### 轻量实验

- 三张 SIM 分别记录：运营商、注册结果、信号强度、PDP 结果、MQTT/TLS 成功率。
- 户外测试 GNSS 冷启动到首次定位时间；室内/窗边记录无 fix 行为。
- 对比“PDP 成功但 MQTT 失败”的可能原因：DNS、证书、端口、安全组、服务器、APN。

### 上手 demo

**Demo S7-1：从 AT 日志看蜂窝联网分层**

目的：把“联网失败”拆成可观察的层级。

步骤：

1. 串口连接 EC800。
2. 依次执行或记录实际固件发出的 AT 流程。
3. 把每条结果按下面层级归类。

记录模板：

```text
SIM 是否就绪：
信号强度：
是否注册运营商网络：
PDP 是否激活：
TCP/TLS 是否连通：
MQTT 是否连接：
失败停在哪一层：
```

你应该理解到：

- SIM 就绪不等于已经上网。
- 注册网络不等于 PDP 已激活。
- PDP 成功不等于 TLS/MQTT 一定成功。

**Demo S7-2：观察 GNSS 冷启动**

目的：理解 GNSS fix 为什么受环境影响很大。

步骤：

1. 带 EC800/GNSS 天线到户外开阔地。
2. 记录开始开启 GNSS 的时间。
3. 记录首次拿到有效经纬度的时间。
4. 回到室内或窗边重复，记录是否无 fix 或耗时明显变长。

记录模板：

```text
地点：
天线位置：
天气/遮挡：
冷启动开始时间：
首次 fix 时间：
经纬度是否有效：
室内/窗边结果：
```

### 验收问题

- `AT` 成功、注册成功、PDP 成功、MQTT 成功分别属于哪一层？
- 为什么 V1 语音走 APP 数据链路，而不是强依赖 VoLTE？
- GNSS 为什么在室内常常不可用？
- TLS 自签 CA 为什么需要嵌入设备或 APP？

## 14. S8：FreeRTOS、低功耗与调试闭环

### 目标

理解嵌入式系统不是“一段程序从上到下跑完”，而是多个任务、事件、中断和电源状态共同工作。

### 项目切入点

阅读：

- [04-固件架构 §2](04-firmware-architecture.md) Task 划分与优先级
- [04-固件架构 §5.2](04-firmware-architecture.md) 电源状态机
- [06-项目执行清单 §P1.6～P1.7](06-project-checklist.md) 低功耗、内存与并发

### 需要理解的概念

- **Task**：RTOS 里的并发执行单元，如 UI、通信、音频、定位、电源。
- **优先级**：紧急任务能更快获得 CPU。
- **队列和事件组**：任务之间传递消息和状态。
- **互斥锁**：保护共享资源，例如 I2C 总线、大块 PSRAM。
- **看门狗**：防止系统卡死无人发现。
- **Light sleep / Deep sleep**：不同深度的省电状态，唤醒源和恢复成本不同。
- **PSM / eDRX**：蜂窝模组侧省电机制，可能影响下行时延。
- **日志分层**：启动日志、驱动日志、业务日志、错误码共同定位问题。

### 轻量实验

- 对照任务表，给每个 task 写一句“它负责什么”。
- 观察亮屏、息屏、Light sleep 进入/退出时日志和电流变化。
- 做一个问题定位模板：现象、复现步骤、当前电源状态、相关外设、日志片段、猜测层级、下一步验证。

### 上手 demo

**Demo S8-1：用日志 + 电流表看电源状态变化**

目的：把 Active、ScreenOff、Light sleep 这些状态和实际电流联系起来。

准备：USB 电流表、串口日志。

步骤：

1. 设备亮屏停在表盘，记录日志状态和电流。
2. 等待自动息屏，记录日志状态和电流。
3. 如果固件已有 Light sleep，等待进入并记录电流。
4. 触摸或按键唤醒，记录恢复到 Active 的日志和电流。

记录模板：

```text
Active 电流：
ScreenOff 电流：
LightSleep 电流：
唤醒方式：
唤醒耗时体感：
相关日志：
```

你应该理解到：

- 低功耗是多个模块一起配合，不只是 CPU 睡眠。
- 屏幕、4G、GNSS 是否关闭/降功耗，会明显影响电流。

### 验收问题

- 为什么跨 Task 通信不能随便直接调用别的 Task 内函数？
- 为什么低功耗不是简单“把 CPU 关掉”？
- 为什么 MQTT 下行时延会受 PSM/eDRX 影响？
- 为什么要记录最小剩余 PSRAM？
