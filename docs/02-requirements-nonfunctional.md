# 02 - 非功能性需求

> 文档编号：NFR
> 适用版本：V1
> 命名规则：`NFR-<类别>-<3位序号>`

非功能性需求约束系统的"质量属性"——它们不直接体现在某个功能上，但会决定整个产品能否真正佩戴在 4 岁孩子的手腕上日复一日地用。

## 2. 物理与人因工程（PHY）

### NFR-PHY-001 重量

- **目标**：整机（含表带）≤ 55 g
- **约束**：4 岁女童手腕承重承受度参考成人 1/3，连续佩戴 6 小时不应有压迫
- **测量**：电子秤
- **应对**：电池 ≤ 4g/100mAh、PCB 4 层薄板、铝合金边框替代不锈钢

### NFR-PHY-002 尺寸

- **目标**：表壳 ≤ 42 mm × 36 mm × 14 mm（长 × 宽 × 厚）
- **表带宽度**：18 mm（通用儿童规格）
- **表带可调范围**：105-135 mm（适配 4-6 岁手腕）
- **测量**：游标卡尺

### NFR-PHY-003 表带与表壳安全

- **要求**：
  - 表带卡扣力 ≥ 30N（手指拽不开，但生产工具可拆）
  - 表带与表壳连接弹簧棒钢制（不锈钢 1.5 mm），拽断力 ≥ 50N
  - 所有外露螺丝为 **Torx T6 防拆头**（家中常见工具拆不开）
  - 表壳外圆角 ≥ R0.3 mm（无尖锐角）
  - 充电触点 4P 不可被孩子误舔短路（涂层 + 几何隔离）

### NFR-PHY-004 防水

- **目标**：IPX5（喷淋水柱可承受）
- **不要求**：IPX7/IPX8（戴去游泳，电子模组寿命会大幅下降）
- **测试**：水龙头中等流速冲洗 5 分钟，不进水
- **应对**：表壳 O 型圈密封、麦克风/扬声器 IPX5 防水网、Type-C/磁吸充电触点防水设计、屏幕黏合用防水胶

### NFR-PHY-005 跌落

- **目标**：1.0 m 高度跌落水泥地，5 次后仍能开机和通话
- **测试**：6 面跌落各 1 次 + 4 角跌落（取 5 次）
- **应对**：表壳硬度 75A 硅胶包边、屏幕全贴合钢化、PCB 板上无超长悬臂元件

### NFR-PHY-006 按键手感

- **要求**：
  - SOS 键凸起 ≥ 0.5 mm，4 岁孩子盲操可定位
  - 按键力 100-150 gf，触发后清脆"咔嗒"反馈
  - 按键寿命 ≥ 50000 次

## 3. 电池与续航（PWR）

### 功耗测量阶段约定（P0 粗测 / P5 精测）

| 阶段 | 深度 | 与需求关系 |
|------|------|------------|
| **P0（原型）** | **子系统粗测**：各工况下电流 **数量级** + **手触温升**；有万用表串电流则记 mA，无则记 **低/中/高** 分档 + 是否烫手 | 及早发现 **异常漏电、模组未休眠、GNSS 常开**；为 [06 - P0.7](06-project-checklist.md) 勾选项 |
| **P5（整机）** | **整机续航**：满电典型场景到关机；与 **NFR-PWR-001** 目标对比 | 自动化或长时间人工记录 |

子系统建议拆分：**屏幕亮 / 息屏**、**4G（驻网待机 / PDP 空闲 / 传数据脉冲）**、**GNSS（搜星中 / 已 fix）**、**录音（I2S 采集中，麦克风到货后）**。P0 不要求与下表预算数值逐 mA 对齐，仅建立 **基线认知**；精测与调策略在 P4～P5 完成。

### NFR-PWR-001 典型使用续航

- **目标**：≥ 24 小时
- **典型使用定义**（与 [00-愿景](00-vision.md) 方案 B「网络语音、非 pstn」一致）：
  - 蜂窝 **数据业务** 约 10 分钟等效负载（MQTT/上推语音等，4G 发射窗）
  - 屏幕亮起 60 分钟（抬腕 + 主动操作）
  - 拍照 5 张
  - 微聊 / 网络语音 10 条（发 5 收 5，每条 8 秒）
  - 位置上报 全天（移动 4 小时 + 静止 20 小时）
- **测量**：P5 阶段自动化测试脚本驱动手表执行上述场景，从满电到关机
- **目标场景的功耗预算**（设计参考，非 P0 硬性对齐）：
  - 深睡（绝大部分时间）：≤ 3 mA
  - 屏幕亮起：≤ 80 mA
  - GNSS 定位中：≤ 50 mA
  - 4G **数据发射窗**（非连续通话）：平均视调制与占空比，峰值仍可达数百 mA（Cat.1 TX）
- **典型占空比与功耗占比（纸面建模，用于与实测对照）**  
  在整机电流不可测前，用 **「平均电流 × 时间占比」** 粗算 24h  mAh，避免只靠体感。以下为 **示例策略**（实现时以固件与 [04-固件架构](04-firmware-architecture.md) `power_manager` 为准，可调）：
  | 机制 | 示例占空 / 频次 | 备注 |
  |------|-----------------|------|
  | MQTT 保活 / 心跳 | 如每 60s 一次短连接，或更长 + 自适应 | 与 EMQX keepalive、运营商省电策略对齐 |
  | GNSS | 如每小时一次短采集 + 移动中加密 | 无 GNSS 料号则改为 LBS |
  | 4G 侧省电 | 模组 **PSM / eDRX**（若芯片与网络支持） | AT 与入网策略见模组手册；可能影响「随时下行」时延 |
  | ESP32-S3 | **Light sleep / Deep sleep**，RTC/GPIO 唤醒，关屏降频 | P1 起做实验性接入，见 **NFR-PWR-006** |
  - **验收**：在 `hardware/docs/power-model.md`（或等价表格）保留一版 **24h mAh 拆分**（建模值 → P0/P5 实测后修订）。
- **电池容量（如 500mAh）与建模**  
  - BOM 中 **503040 等 ~500mAh** 为 [03-硬件设计](03-hardware-design.md) 目标；**须在 P4 定型前** 用 **上表占空比 + NFR-PWR-001 典型日** 做一次 **mAh 求和**（含 ≥20% 余量），证明 **不显著低于 24h 目标**。  
  - P0 子系统粗测（[06 - P0.7](06-project-checklist.md)）提供 **各状态电流输入**；若粗算已明显不足，提前考虑 **减屏时、加大电池、eDRX** 等，避免整机打样后才发现续航不够。

### NFR-PWR-002 待机续航（用于演示）

- **目标**：≥ 72 小时（充满电后不使用，仅维持 MQTT 心跳 + 抬腕检测）
- **测量**：放置 3 天

### NFR-PWR-003 充电时间

- **目标**：0% → 100% ≤ 2 小时
- **充电方式**：磁吸 4 触点（VBUS + GND + D+ + D-，CC1.5 速率），5V/1A
- **管理芯片**：SY6970
- **过充保护**：电池电压超 4.25V 切断
- **温度保护**：电池温度 > 45°C 暂停充电

### NFR-PWR-004 低电量行为

- **要求**：
  - 电量 < 20%：表盘电量图标变红，APP 推送
  - 电量 < 10%：自动关闭非关键功能（拍照、抬腕亮屏），仅保留 **网络语音联系** 与 SOS
  - 电量 < 5%：自动保存状态后关机，留 1% 给 RTC

### NFR-PWR-005 电池循环寿命

- **目标**：500 次充放电后容量 ≥ 80%
- **应对**：充电截止电压设为 4.2V 而非 4.25V，牺牲 ~5% 初始容量换寿命

### NFR-PWR-006 低功耗策略与实现阶段

- **要求**：将下列策略 **纳入固件规划**，并在 **P1 末 / P2 初**至少完成 **可测量实验**（能开能关、能记电流差），完整策略在 **P4～P5** 与结构一起收束。  
  | 层级 | 策略 | 最低实现阶段建议 |
  |------|------|-------------------|
  | ESP32-S3 | GPIO/定时器唤醒、Light sleep、（可选）Deep sleep 下挂关键外设 | P1～P2 实验 |
  | 蜂窝模组 | **PSM、eDRX**、关闭 GNSS 不定位时、PDP 按需激活 | P1～P2 与 AT 驱动同迭代；以模组 AT 与入网实测为准 |
  | 业务层 | 位置上报 **移动加密/静止稀化**、MQTT **可配置间隔**、上课禁用关屏等 | P2～P3 |
- **与清单对应**：功耗见 [06 - §P1.6](06-project-checklist.md)，内存与并发见 [06 - §P1.7](06-project-checklist.md)。

## 4. 性能与响应（PERF）

### NFR-PERF-001 UI 帧率

- **目标**：表盘、列表滚动 ≥ 30 FPS；过场动画允许 20 FPS
- **测量**：LVGL 内置 FPS 计数
- **应对**：QSPI 屏 DMA 双 buffer、LVGL 部分刷新模式、复杂动画关闭抗锯齿

### NFR-PERF-002 关键操作响应时延

| 操作 | 目标时延 | 测量定义 |
|---|---|---|
| 抬腕到亮屏 | ≤ 300 ms | IMU 中断到屏幕显示首帧 |
| 点击表盘到主菜单 | ≤ 500 ms | 触摸事件到菜单完整渲染 |
| 长按 SOS 到开始拨号 | ≤ 8 s（信号良好）| 按键到 ATD 命令发出 |
| 拨号到对方电话振铃 | ≤ 6 s | ATD 到 CALL CONNECT 事件 |
| 点击微聊录音按钮到开始录音 | ≤ 200 ms | 触摸到 I2S 开始采集 |
| 点击照片到全屏展示 | ≤ 500 ms | 触摸到 JPEG 解码完成 |
| MQTT 命令下行到手表执行 | ≤ 3 s（信号良好）| APP 发出到手表 ACK |
| APP 启动到首页 | ≤ 1.5 s（暖启动）| App icon 点击到地图渲染 |

### NFR-PERF-003 启动时间

- **冷启动**：电源键 → 表盘 ≤ 8 秒
- **唤醒**：从深睡到响应触摸 ≤ 200 ms

### NFR-PERF-004 后端 API 响应

- **目标**：P95 ≤ 200 ms（同城云服务器，4G 良好网络）
- **承载**：单家庭 1 台手表 + 4 家长 APP，QPS 峰值 < 10
- **当前架构余量**：96 核分片 + 31G 内存，承载 1000 台手表无压力

## 5. 可靠性与容错（REL）

### NFR-REL-001 4G 信号波动韧性

- **场景**：地下停车场、高铁、电梯
- **要求**：
  - 信号丢失 < 5 分钟自动重连不丢消息
  - 信号丢失 ≥ 5 分钟进入"离线缓存"模式，本地存事件，重连后批量上报
  - MQTT keepalive 60s，断线后指数退避重连（1s, 2s, 5s, 10s, 30s, 60s）

### NFR-REL-002 软件看门狗

- **要求**：
  - ESP32 双核 IDLE 任务喂狗周期 5 秒
  - 应用任务每 30 秒喂一次专用看门狗
  - 看门狗超时 60 秒自动重启
  - 重启原因（panic、看门狗、断电）写入 NVS，开机回传后端

### NFR-REL-003 故障告警

- **要求**：以下事件 1 分钟内推送家长 APP
  - 设备离线 > 30 分钟（且非夜间充电时段）
  - 电池电量 < 10%
  - SIM 卡丢失 / 无法注册网络
  - GNSS 模块失效（连续 3 次定位失败）
  - 连续 3 次软件崩溃

### NFR-REL-004 数据丢失容忍

- **要求**：
  - 位置数据：丢失 < 1% （非关键，丢了下次再传）
  - 通话记录：0 丢失（NVS 持久化）
  - 微聊消息：0 丢失（发送端保留到 ACK 收到）
  - 照片：失败重试 3 次，仍失败 APP 提示用户

### NFR-REL-005 服务器单点故障

- **现状**：V1 单机部署，单点故障窗口存在
- **应对策略**：
  - 每日凌晨自动备份 PostgreSQL + MinIO 到本地另一磁盘
  - 每周备份导出到本地 NAS（rclone）
  - V2 评估升级到主备或异地多活

## 6. 安全性（SEC）

### NFR-SEC-001 传输层安全

- **要求**：
  - 所有手表 ↔ 后端通信走 TLS 1.2+（手表端 mbedTLS、后端 Caddy）
  - HTTPS API 端口 18443、MQTT over TLS 端口 18883
  - 自签 CA 证书钉扎（pinning），手表/APP 内置 CA 公钥
  - 服务器证书有效期 5 年，CA 证书 10 年
  - 证书续签前 90 天告警，OTA 推送新证书

### NFR-SEC-002 设备身份认证

- **要求**：
  - 每台手表出厂烧录唯一 `device_id`（基于 ESP32 efuse 派生）+ Ed25519 私钥
  - 后端存对应公钥，所有请求需签名
  - JWT `device_token` 30 天过期，刷新走签名挑战
  - 设备私钥存 ESP32 加密分区，不可读出

### NFR-SEC-003 家长账户安全

- **要求**：
  - 登录采用手机号 + 短信验证码（不存密码）
  - 异地登录新设备需短信二次验证
  - JWT `user_token` 30 天，可在 APP 端"安全中心"主动登出所有设备
  - 解绑设备需主家长 + 短信二次验证（防恶意解绑）

### NFR-SEC-004 OTA 固件签名

- **要求**：
  - 所有 OTA 包用 Ed25519 私钥签名
  - 公钥写入 bootloader，不可改
  - 验证失败的 OTA 包丢弃，记录告警
  - bootloader 启用 secure boot（V1.1 引入，V1 可先关闭以便调试）

### NFR-SEC-005 防中间人攻击

- **要求**：
  - 证书钉扎（pinning）实现强制，无降级到系统 CA
  - 任何证书不匹配立即断开 TCP 连接并指数退避重连
  - 后端 access log 记录每次 TLS 握手指纹用于审计

### NFR-SEC-006 物理篡改防护（轻度）

- **要求**：
  - 表壳 Torx 防拆螺丝
  - ESP32 关闭 JTAG（生产固件烧录前 efuse 烧 `DIS_USB_JTAG`）
  - UART 调试口在生产固件中关闭打印（仅 panic 日志保留）

## 7. 隐私与数据保护（PRIV）

### NFR-PRIV-001 数据最小化

- **原则**：只收集"实现功能所必需"的数据，不收集任何额外维度
- **明确不收集**：
  - 设备唯一标识符之外的硬件信息（MAC 地址、IMEI 等）
  - 周边 WiFi BSSID（V1 不做 WiFi 定位，V2 谨慎评估）
  - 蓝牙周边设备列表
  - 任何用于"用户画像"的行为聚合数据

### NFR-PRIV-002 数据加密静态存储

- **要求**：
  - PostgreSQL 中存储的位置、消息、通话记录使用 PG 自带 `pgcrypto` 列级加密（敏感字段）
  - MinIO 中的录音、照片使用 SSE-S3 服务端加密
  - 密钥存独立的 `secrets/` 目录，备份独立于业务数据

### NFR-PRIV-003 数据保留周期

| 数据类型 | 保留期 | 过期处理 |
|---|---|---|
| 实时位置 | 90 天 | 自动删除 |
| 通话录音 | 不保留 | 通话不录音 |
| 微聊语音 | 180 天 | 转冷归档 1 年后删除 |
| 照片 | 永久（仅本家长可访问） | 家长 APP 中可手动删除 |
| 通话记录元数据 | 1 年 | 自动删除 |
| 服务器访问日志 | 30 天 | 自动滚动 |

### NFR-PRIV-004 数据访问授权

- **原则**：仅家庭组成员能访问该家庭设备数据
- **后端实现**：所有查询带 `family_id` 过滤，跨家庭查询直接拒绝
- **审计**：所有"访问敏感数据"操作（位置/录音/照片）写访问日志，APP 中"隐私设置"可查看

### NFR-PRIV-005 第三方依赖审计

- **禁用**：神策、友盟、Bugly、极光推送、Firebase Analytics、Sentry 等任何用户行为数据上报
- **允许**：
  - 高德地图 SDK（仅地图渲染，不上报用户位置）
  - APNs / 厂商推送通道（仅推送通道，不发送敏感正文）
  - 阿里云 / 腾讯云短信 API（仅登录验证码）
- **审计方式**：APP 打包前用 `mitmproxy` 抓包，确认无意外的对外请求

### NFR-PRIV-006 数据导出与删除（用户权利）

- **要求**：
  - APP 中"我的 > 导出全部数据"可下载 JSON 包（自己孩子的全部数据）
  - APP 中"我的 > 注销账户"删除全部数据（不可逆，二次确认）
  - 注销执行：物理删除 PG 中相关 family_id 数据、MinIO 中相关对象

## 8. 可维护性（MAINT）

### NFR-MAINT-001 代码质量

- **要求**：
  - 固件：编译警告 0、`-Wall -Wextra`、关键模块单元测试覆盖 ≥ 50%
  - 后端：`go vet` + `staticcheck` 通过、单元测试覆盖 ≥ 60%
  - APP：`flutter analyze` 0 warning、Widget 测试覆盖 ≥ 40%

### NFR-MAINT-002 日志规范

- **格式**：结构化 JSON（后端、APP）+ 简化 KV（手表端）
- **级别**：DEBUG / INFO / WARN / ERROR / FATAL
- **生产开关**：手表端默认 INFO、后端默认 INFO、APP 默认 WARN
- **远程日志**：手表 ERROR 级以上自动上报到后端

### NFR-MAINT-003 监控与告警

- **后端**：
  - Prometheus 抓取 Go API、EMQX、PostgreSQL、MinIO 指标
  - Grafana 仪表盘：QPS、P95 时延、错误率、MQTT 连接数、磁盘占用
  - 关键告警：错误率 > 1%、磁盘占用 > 80%、MQTT broker 断开 → 邮件 + 短信
- **手表端**：
  - 崩溃 / 看门狗重启自动上报后端
  - 后端聚合统计，每周报表

### NFR-MAINT-004 文档与代码同步

- **要求**：
  - 公共 API 变更需更新 `docs/05-cloud-and-app.md`
  - 硬件原理图变更需更新 `docs/03-hardware-design.md`
  - 每个 release tag 关联完整文档快照

### NFR-MAINT-005 部署自动化

- **要求**：
  - 后端 Docker 镜像 GitHub Actions 自动构建
  - 部署：`docker compose pull && docker compose up -d`
  - 数据库迁移：`migrate up` 自动执行（启动 hook）
  - 回滚：保留前 3 个版本镜像

## 9. 可观测性（OBS）

### NFR-OBS-001 链路追踪

- **要求**：
  - 后端用 OpenTelemetry，每个请求生成 trace_id
  - APP 日志附带 trace_id
  - 手表关键事件携带 trace_id（如 SOS 触发流程）
  - Jaeger / Tempo 后端查询（V1.1 引入）

### NFR-OBS-002 关键事件埋点

- **手表端事件**：
  - 开机、关机、断网、重连、SOS、电量阶段（80/50/20/10/5%）
  - 看门狗触发、panic、OOM
- **后端事件**：
  - API 请求、MQTT 上下行、设备上下线
- **APP 事件**：
  - 启动、登录、关键操作（拨打、位置查询）
- 所有事件结构化记录，便于后续分析

### NFR-OBS-003 现场调试支持

- **要求**：
  - 手表内置"开发者模式"（隐藏，连续点击表盘 10 次唤醒）
  - 开发者模式可看：固件版本、信号强度、RAM/Flash 占用、最近 50 条日志
  - 4G 模组 AT 命令直接透传（开发期）

## 10. 合规与法律边界（LEGAL）

### NFR-LEGAL-001 商用与认证边界

- **声明**：本项目为**个人自用项目**，不进行任何商业销售、不公开发售样机
- **不涉及**：
  - CCC / SRRC / CTA（手表整机型号核准）
  - FCC / CE（出口认证）
  - 食品级硅胶 LFGB / FDA（表带）
- **建议保留**：BOM 中模组本身的认证文档（EC800N 已通过 SRRC/CTA）

### NFR-LEGAL-002 SIM 卡实名

- **要求**：
  - SIM 卡使用父母身份证实名办理消费级儿童副卡
  - 不使用任何"灰色渠道物联网卡"
- **风险**：实名信息由运营商管理，不在本系统存储

### NFR-LEGAL-003 个人信息保护（自用边界）

- **依据**：《个人信息保护法》《未成年人保护法》《未成年人个人信息网络保护规定》
- **自用场景豁免**：用于家庭内部、不公开提供服务，保护要求适度但不可放松
- **采取的措施**：
  - 所有敏感数据本机加密 + 传输加密
  - 不共享给任何第三方
  - 监护人作为数据控制者，全权决定数据处置

### NFR-LEGAL-004 开源协议

- **代码**：MIT License（硬件原理图、固件、后端、APP）
- **不开源**：CA 私钥、设备私钥、用户数据
- **第三方依赖**：保留所有上游许可证声明

## 11. 环境与温度（ENV）

### NFR-ENV-001 工作温度

- **目标**：-10°C 到 +45°C 正常工作
- **测试**：
  - 冰箱 -10°C 放置 1 小时后开机使用
  - 阳台夏季 +45°C（避免阳光直射后）放置 2 小时
- **失效模式**：超温自动关机保护

### NFR-ENV-002 存储温度

- **目标**：-20°C 到 +60°C
- **应对**：锂电池温度范围决定上限

### NFR-ENV-003 湿度

- **目标**：相对湿度 5%-95%（无凝露）

### NFR-ENV-004 EMC

- **关键考虑**：
  - 4G 模组发射功率 ~2W，距离屏幕和 IMU 很近，必须屏蔽
  - 应对：4G 模组下方铺地、屏蔽罩、关键信号线远离 RF 走线 ≥ 2mm
