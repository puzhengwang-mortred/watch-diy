# 00 - 愿景与范围

> 文档编号：VISION
> 适用版本：V1（2026 Q2 – Q3）
> 维护者：项目所有者

---

## 1. 项目背景

为 4 岁女儿打造一只完全 DIY 的电话手表。市面上的小天才/华为/360 等产品体验已经很好，但本项目的核心动机不是"做得更好"，而是：

- **掌控隐私数据**：孩子的位置、录音、照片不希望经过任何第三方云
- **了解每一行代码**：从主板原理图到 APP 的最后一行代码，全部出自自己之手
- **作为一个长期可玩的家庭项目**：未来还能在上面加 AI 助手、家庭物联网等扩展

## 2. 目标用户与使用场景

### 2.1 主要用户

- **佩戴者**：4 岁女童，手腕周长 ~105 mm，词汇量 1000+，能识别少量汉字与图标
- **管理者**：女儿的爸爸与妈妈（项目所有者及配偶）
- **辅助使用者**：祖父母（仅作为通话来电方与定位查看人）

### 2.2 典型使用场景（按使用频次降序）

1. **日常语音联系**：放学路上孩子用 **网络语音/微聊** 问家长「几点到家」（依赖 4G 数据与 APP）
2. **位置查询**：家长在 APP 中查看孩子当前位置 / 历史轨迹
3. **微聊语音**：孩子按住手表录 5–10 秒语音"我饿了" → 妈妈手机收到推送
4. **拍照**：孩子在幼儿园看到喜欢的画拍下来分享给妈妈
5. **SOS 求助**：孩子遇到陌生人/迷路时长按 SOS，**家长 APP 强提醒 + 位置上报**（可选短信兜底）
6. **闹钟**：早上 7:00 起床、下午 5:30 收玩具

### 2.3 非目标用户

- 不面向 8 岁以上儿童（他们的需求是社交、游戏、学习，本项目刻意不做）
- 不面向商业销售（不做包装、说明书、售后体系、CE/FCC 认证）

## 3. 项目原则（取舍准则）

排序即优先级，遇到冲突时优先保前者：

1. **安全 > 功能 > 性能 > 美观**：电池过充防护、表带不可拆卸吞食、表壳无锐角 > 多一个炫酷功能
2. **隐私自主 > 用户体验便利**：宁可不接微信，不接入任何第三方 SDK（神策/友盟/极光/Bugly 等都不要）
3. **可维护 > 极致**：单台调试方便比量产美感更重要（V1 范围）
4. **公开设计、私有数据**：硬件/固件/服务端代码 MIT 开源，但 CA 私钥、用户数据严格私有
5. **少即是多**：每加一个功能都要问"是不是 4 岁孩子真的需要"

## 4. 范围划定

### 4.0 V1 语音方案变更（2026-05，已采纳）

**原方案 A**：运营商 **VoLTE**，手表与家长手机 **pstn 直拨**（最像小天才，模组与资费要求高）。

**现方案 B（当前 V1）**：不强制蜂窝电话。在 **4G/蜂窝数据可用** 时，孩子与家长通过 **自建后端 + MQTT/HTTPS + APP** 收发 **语音消息或半双工语音**（技术上延续「微聊」栈：Opus 分段上传/下发；若要做「按住即说、松手即停」的伪实时，可在同栈上加短间隔分片或低延迟通道）。

**方案 B 带来的取舍（接受即写进需求）**：

- **无网即无「语音联系」**：地下室、隧道、欠费、仅 2G 等场景，不能像电话一样保底；缓解：缓存未发消息、出网后重试、SOS 用 **推送 + 短信（若模组支持且你愿意开）** 兜底。
- **家长侧体验依赖 APP**：不能像来电一样系统级响铃（除非后续做 VoIP/CallKit 深度集成，复杂度高）；依赖 **推送 + 强振动 + 全屏告警页**。
- **蜂窝模组选型放宽**：**Cat.1 数传类**（如 EC800KCNLC 数传核心板）可作为 P0/P1 验证与 V1 候选，**不必**为 VoLTE 单独锁 EC800N 语音订货号；**仍建议带 GNSS 的订货号** 以满足定位。
- **SOS**：不再定义为「必须 pstn 依次拨号」；改为 **强提醒链路**（推送全员 + 上报位置 + 可选自动发一条 **短信** 给主家长 + 手表端反复播放告警音）。若你仍希望「能打通手机」，可作为 **V1.1 加回** 或保留为可选硬件能力。

后续文档中「通话」类条目以 **网络语音 / 微聊增强** 为准；`FR-CALL-*` 在 [01-功能性需求](01-requirements-functional.md) 顶部有变更说明。

### 4.1 V1（MVP）必须做

| 模块 | 能力 | 备注 |
|---|---|---|
| 语音联系 | **蜂窝数据下** 与家长 APP **双向语音消息**（含按住说话、推送提醒）；**不强制** VoLTE pstn | 与 §4.0 方案 B 一致；白名单逻辑可在 APP 侧实现 |
| 定位 | 实时 GPS/BDS 定位 + 历史轨迹 | 室外 5m 精度，室内基站辅助 |
| SOS | 长按触发 **强提醒**（推送 + 位置上报 + 本地告警音）；**不强制** pstn 拨号 | 可选短信兜底见 §4.0 |
| 微聊 | 按住录音 / 语音消息；可与「语音联系」合并实现 | Opus 编码 ≤16kbps |
| 拍照 | 200 万像素静态照片 | 自动上传 APP 相册 |
| 表盘 | 1 个默认卡通表盘 + 时间 / 电量 | 圆形 466×466 |
| 计步 | 当日步数 / 周报 | QMI8658 板载 |
| 闹钟 | 单次/重复闹钟，振动 + 铃声 | 最多 5 个 |
| 电子围栏 | 离开/进入家/学校区域时家长 APP 告警 | 圆形围栏，半径可调 |
| 上课禁用 | 在指定时间段内禁用 **微聊/语音联系** 等非紧急能力 | **SOS 始终可用** |
| 充电 | 磁吸触点 4P，5V/1A | 2 小时充满 500mAh |
| 家长 APP | iOS + Android | Flutter 单代码库 |
| 后端 | 自建（云服务器单机 Docker Compose） | 裸 IP + 自签证书 |
| OTA | 固件远程升级 | 双分区 A/B，差分 + 全量 |

### 4.2 V2（V1 完工 + 实际佩戴 1 个月后启动）

- 运营商 **VoLTE pstn 直拨**（与方案 A 对齐，可选增强）
- 视频通话（要 Cat.4 模组 + 后端 SFU + 摄像头方向反转）
- 心率监测（MAX30102，可选血氧）
- NFC 校园卡/公交卡（PN532）
- 多表盘自定义
- AI 大模型对话助手（接入小模型，端侧 + 后端混合）
- 跌倒检测（IMU 算法 + AI）

### 4.3 V3 及更远

- 学习卡片（拼音 / 简单算术）
- 家庭其他成员手表互联（祖父母 / 配偶手表互看位置）

### 4.4 明确不做（永远不会做）

| 不做的事 | 原因 |
|---|---|
| 应用商店 / 装第三方 APP | 安全、合规、平台审核负担巨大 |
| 短视频、游戏 | 与"安全设备"定位冲突 |
| 支付（NFC 钱包外） | 涉及金融合规，自用不值得 |
| 接入第三方分析/广告 SDK | 与"隐私自主"原则冲突 |
| 海外发售 / 认证（FCC/CE） | 自用项目，不商用 |
| 鸿蒙/Android Wear 上构建 | 自研 RTOS 可控性更高 |
| 公开测试版给陌生人安装 | 隐私边界仅限自己家庭 |

## 5. 关键决策快照

> 详细论证见对应章节，此处仅汇总结论。决策一旦确认，未来更改必须有强理由。

| 决策项 | 选择 | 理由（摘要） | 章节 |
|---|---|---|---|
| 通信能力 | 4G Cat.1 **数据** + WiFi + BLE + GNSS（VoLTE pstn **不强制**） | V1 采用 §4.0 方案 B | 00/03 |
| DIY 程度 | 模块化 DIY（成熟模组 + 自研载板） | 个人能力可控、3-6 个月可交付 | 03 |
| 主控 | ESP32-S3-WROOM-1U-N16R8 | 16M Flash+8M PSRAM、生态成熟、LVGL 友好 | 03/04 |
| 4G+GNSS 模组 | 移远 EC800 **系列**（含 EC800K/M/N 等 **数据 + GNSS** 订货号） | 数传版可接受；以 PDP 稳定 + GNSS 为准 | 03 |
| 屏幕 | 1.43" 圆形 AMOLED 466×466 SH8601 | 童趣造型、低功耗（黑底）、生态文档丰富 | 03 |
| SIM | Nano SIM 卡座 + 运营商儿童副卡 | 个人可办、可换卡测试 | 03 |
| 视频通话 | V1 不做，V2 再加 | 后端 SFU 复杂度高 + Cat.1 带宽不足 | 00 |
| 后端形态 | 国内云服务器 + 裸 IP + 自签证书 + 高位端口 | 省域名年费 + 省 ICP 备案等待 | 05 |
| 后端栈 | Go + Gin + EMQX + PostgreSQL + MinIO + Caddy | 性能/生态/单机部署友好 | 05 |
| 部署方式 | 单机 Docker Compose | 31GB 内存绰绰有余，运维简单 | 05 |
| 家长 APP | Flutter 3.x | 一码 iOS+Android、热重载开发快 | 05 |
| 数据加密 | 静态 AES-256-GCM + 传输 TLS 1.2+ | 儿童敏感数据兜底 | 02 |
| 推送通道 | iOS APNs + 国内 Android 厂商推送（小米/华为/OPPO/vivo） | FCM 国内不可达 | 05 |

## 6. 成功指标（V1 验收）

V1 视为成功的客观条件：

| 指标 | 目标值 | 测量方式 |
|---|---|---|
| 4 台样机焊接全部上电成功 | 4/4 | 上电后 ESP32 串口能打印启动日志 |
| 三家运营商 **蜂窝数据**（PDP + MQTT/HTTPS）可用率 | ≥ 2/3 | 各卡各测 10 次业务请求成功 |
| 室外定位收敛时间 | ≤ 60 秒（冷启动） | 在家附近空地实测 |
| 室外定位精度 | ≤ 10 米 | 与手机 GPS 对比 |
| 续航（典型使用：每天 **网络语音/微聊合计 10 分钟** + 屏幕亮 1 小时） | ≥ 24 小时 | 满电到关机实测 |
| 微聊端到端时延 | ≤ 5 秒（4G 良好） | 手表录完到 APP 收到推送 |
| SOS 触发到 **家长 APP 收到告警**（推送或 MQTT 可达） | ≤ 8 秒（信号良好） | 长按到服务端确认入库且推送已发出 |
| 重量 | ≤ 55 g | 电子秤 |
| 总厚度 | ≤ 14 mm | 游标卡尺 |
| 实际佩戴 7 天无故障 | 4/4 台 | 女儿轮换佩戴 |

## 7. 风险与假设

### 7.1 关键假设

- EC800 系列在我所在城市运营商网络下 **数据业务**（PDP/MQTT）稳定（**待验证**，P0 任务）
- ESP32-S3 在 **LVGL + 蜂窝 + GNSS + 录音/网络上传** 并发下 **不发生内存耗尽**（**待验证**）：模块级 PSRAM 预算、互斥与抢占规则见 [04 - §3.2 / §3.5](04-firmware-architecture.md)（含 **录音优先时暂停 GNSS 新一轮捕获** 等策略）；P1 做 `heap_caps` 长时压测
- 嘉立创 4 层板 + SMT 打样质量稳定（基于其他项目历史经验，可信度高）
- 父母身份证办理的消费级 SIM 在手表形态下 **流量套餐** 可用、不限速到不可用（仍待实测）

### 7.2 主要风险

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| 自研 PCB 在 4G 发射时干扰其他电路 | 中 | 高（设备无法稳定通信） | 4 层板 + 完整地平面 + EMC 屏蔽罩 + 资深朋友 review |
| 续航不达标 | 高 | 中（用户体验降级但不致命） | 提前测各子系统功耗、多档电池备选 |
| 摄像头与 4G 模组共享 PSRAM 带宽冲突 | 中 | 中（拍照失败） | 用 SD 卡缓存中间帧、降低拍照分辨率 |
| ESP32-S3 跑 LVGL 9 帧率不达标 | 低 | 中（UI 卡顿） | 已知大量项目跑通，最坏降级 LVGL 8 |
| 自签证书路线被云厂商扫描封端口 | 低 | 高（服务下线） | 预案：1 天买域名 + 提交备案，期间走 IP |
